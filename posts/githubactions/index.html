Hugo + Github Actions

2021-06-06 320 words 2 minutes
Contents

I’ll admit, I was having a little difficulty understanding how to organize and deploy my first hugo site. Every time I pushed changes, Github Pages wasn’t deploying the site the way I was expecting. I missed a small but critical step.

In the post I referenced in my first post hello hugo, I missed the step to cd public then push. I was pushing the entire project and hoping there was some hugo magic that would work under the hood to render the site once deployed. It was so easy to hugo server -D and make changes and see them update immediately. Why wasn’t it working for my deployments? I kept have build errors.

Luckily, in my frustration to deploy the various themes successfully via Github Pages to see which one I liked best, I ended up reading more hugo docs and found another post which proposed to solve my problem in 30 min or less, no less.

I realized the obvious which is that hugo is only generating a static site and Github Pages is only serving a static site. My confusion was related to when I pushed changes the other day and deleted the original site. Github Pages was rendering the README.md as index, and I liked the clean and simple site equating it to hugo, when in reality it was just a pure static .md file.

The problem could have been solved if Github Pages could select a folder other than /root or /docs, but that’s just me.

Now I have a workflow that works for me. I created a Github Actions workflow and separated my site source from the publicly deployed Github Pages site. Overall that give me the control to keep the content public, but the source control in a public or private repo.

Hugo is definitely growing on me and I’m excited to look into the templating and potentially create a simple custom theme.
